# コンセプト設計メモ(別セッションの記録・原文)

> 2026-07-17、別セッションでの対話記録をそのまま保存したもの。
> このセッションの決定事項との突き合わせは [design-agenda.md](design-agenda.md) を参照。

## コアコンセプト

- AIにとって書きやすく、チームにとって読みやすいシンプルな言語
- AIの「書きやすさ」の本質は見た目の簡潔さではなく、以下にあると整理した:
  - 一つのことをやる書き方が一つしかない(Goの哲学)
  - ローカルに推論できる(明示的な型注釈は「圧縮されたコンテキスト」)
  - エラーメッセージが修正方法を示す(エージェントの自己修正ループと相性が良い)
  - 学習データとの近さ(TS/Go風の見た目なら既存知識を転用できる)
- 設計の判断軸: 「AIの生成ミスを機械的に弾く」「ファイル単体で文脈が完結する」

## 決定事項

### 構文の基本方針

- インデントベース構文は不採用。明示的なデリミタ(波括弧等)を使う
  - 理由: AIはdiff/部分編集でコードを書き換えることが多く、空白が構文的意味を持つと
    インデントのズレが即バグになる。明示的デリミタなら機械的に検証可能

### 分岐

- switch文ではなくRust風のmatch式を採用
  - 網羅性チェック(ケース漏れがコンパイルエラー)
  - 式として値を返せる(一時変数・breakが不要になり宣言的)
  - ただしRustのパターンマッチをフル移植はしない。網羅性チェック + 式 + ガード(if)程度に絞り、
    深いネストの分解は段階を分けて書かせる

### 型システム

- enumは不採用、union型を採用
  - リテラル型のユニオン(`"active" | "paused" | "done"`)と判別可能ユニオン
    (discriminated union)で閉じた集合を表現
  - タグ付きオブジェクトのユニオン(`{ kind: "ok", value: T } | { kind: "err", error: E }`)で
    Rustのデータ付きenum相当を実現
  - 選定理由: 概念が増えない / AIの学習データが豊富(TS) / JSONへのシリアライズが透過的
- null無し: `T | none` のようなunionで表現し、matchやif narrowingで安全に取り出す
- matchの網羅性チェックはunionと連動させる(三点セット: union + 判別可能ユニオン + 網羅性チェック)

### エラー処理(検討中・方向性は決定)

- 例外なし、エラーは値(Result的なunionを返す)
  - 例外は「見えない制御フロー」でAIのcatch漏れを検出できないため
- 未決: Rustの `?` のような伝播糖衣を入れるか
  - 入れると連鎖処理が短くなるが「見えない早期リターン」が復活する
  - レビュー最優先なら match + 早期リターンのみに絞る割り切りもあり

### 実行環境

- フロントエンド・バックエンド両対応
- コンパイルターゲットはJS
- 型をフロント/バックで共有し、APIコントラクトのズレをコンパイルエラーにする構想
  - TSがtRPC/Zodで後付けしていることを言語ネイティブで実現
  - 判別可能ユニオンがJSONにそのまま乗るので変換層がほぼ不要

### 非同期・並行処理

- async/awaitではなくGo風の仕組みを採用(goroutine相当)
- channel(ch)も採用予定
- 利点:
  - 関数の色問題(function coloring)が消え、`await`付け忘れバグが原理的に存在しなくなる
  - 制御フローが上から下に素直に読める(レビューしやすい)
- JS上の実装に関する注意点(検証項目):
  - generator/CPS変換による出力コードサイズとスタックトレース(source map)の品質
  - Promiseベースのnpmライブラリとのブリッジを言語仕様として定義
  - JSはシングルスレッドなので並行(IO)には効くが並列(CPU)にはならない → ドキュメントで割り切りを明記
- channel設計の論点:
  - 型付きchannel(`ch<User>`)は必須
  - Goの罠(unbuffered/bufferedの挙動差、close忘れ・二重close)は簡略化の余地あり
    - 例: closeをスコープと紐づける、受信を `T | closed` のunionで返してmatchで処理(null無し設計と一貫)
  - select相当は初期は見送りか、入れるならmatchに構文を寄せる
  - シングルスレッドゆえchannelは「排他」でなく「データフローの構造化」の道具と位置づける

## 提案済み・未決のアイデア

- デフォルト不変、可変は明示(`let` / `mut`): レビューで `mut` だけ注視すればよくなる
- 公式フォーマッタ一択・設定なし(gofmt方式): AIの出力が安定し、diffが実質的な変更だけになる
- 公開関数の型注釈を必須に: シグネチャは「本体を読まずに使える」圧縮されたコンテキスト
- 名前付き引数: `createUser(name: "x", age: 3)` 形式。AIの引数順ミスを防ぎ自己文書化される
- コンパイラに構造化出力モード: エラーをJSONで吐くフラグ。エージェントのエラー→修正ループを高速化
- 標準ライブラリの層分け: coreは両環境共通、DOM操作やファイルI/Oは環境別モジュールに分離。
  importで環境を機械的に判別し、サーバーコードで `document` を触る等のミスをコンパイル時に弾く
- フロントでUIイベントをchannelとして扱う統一感の可能性

## コード例(構文は仮)

```
// shared/api.t — フロント/バック共有の型定義
type GetUserResponse =
    { kind: "ok", user: User }
  | { kind: "notFound" }
  | { kind: "unauthorized" }
```

```
// null無し + match
fn findUser(id: string): User | none {
    ...
}

let user = findUser(id: "u123")

match user {
    User => sendMail(to: user.email),
    none => log("user not found"),
}
```

```
// if narrowing + 早期リターン
let user = findUser(id: "u123")
if user is none {
    return none
}
// この行以降、userはUser型に絞り込まれている
sendMail(to: user.email)
```
