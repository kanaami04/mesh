## Task 04 で Mesh が踏んだ2つの往復(最重要の収穫)

**往復1: 複数行unionの行頭`|`継続が書けない(負転移・言語側の穴)**
被験体はTSの定番フォーマットで判別可能unionを複数行に折った:

```
type Expr = { kind: "num", value: int }
    | { kind: "add", left: Expr, right: Expr }
    | ...
```

Meshは改行を文末とみなすため1行目で宣言が完結し、`use 'struct Expr { ... }' to define a
data shape` エラーになった。**union自体の理解は完璧で、フォーマットだけで落ちた**。
TSは行頭`|`継続を許すので、これはF-6系(TS学習データとの負転移)のフォーマット版。
→ **✅ 2026-07-20対応済み**: パーサが型宣言中の「改行(ASIの`;`)の先の`|`」を継続として
読むようになり、行頭`|`・行末`|`の両フォーマットが動く。カードの判別可能union例も
複数行形に変更して実例化した

**往復2: 関数名 `eval` が予約語**
仕様が関数名を`eval`と指定していたため、`RESERVED`(JS strict mode由来、カード実験第6回で
追加)に当たった。エラーメッセージ(`'eval' is a reserved word and cannot be used as a name`)
から1往復で`evaluate`へ全箇所リネームして解決 — 「修正方法が分かるエラー」は機能した。
→ **✅ 2026-07-20対応済み**: RESERVED全34語をカードのBindings節に明記した。
⚠️ 計測の正直な注記: TS被験体のプロンプトにだけ「evalが予約されていれば別名で可」の
注記があった(TSもstrict modeで`eval`を関数名にできないため防衛的に入れたもの)。
Mesh/Goには無し。Goは`eval`が有効なので影響なし、Meshはこの注記があれば往復2は
起きなかった可能性がある — 次回はプロンプトを完全対称にすること

**Mesh 02の往復1: match の「何もしない」アーム(新発見の負転移)**
Haikuは `match port { none => {}  _ => print(...) }` とRust風の空ブロックアームを書いた。
Meshのmatchは式なのでブロックは書けず`unexpected '{'`。修正は`if !(port is none) { ... }`
への書き換え(F-6のド・モルガン形が復帰経路として機能)。
→ **✅ 同日対応済み**: カードのmatch節に「アーム本体は式のみ・何もしないアームは無い —
その場合はifを使う」を明記した

**Mesh 04 / TS 04の往復1: 対称プロンプトでは両者とも`eval`で落ちた**
プロンプト対称化の結果、TSも`TS1100: Invalid use of 'eval' in strict mode`で落ちた —
**第1RでTSのTask 04が一発だったのはプロンプトの注記のおかげだった**ことが裏付けられた。
Meshは新カードに予約語一覧があるがHaikuは読み飛ばした(Sonnetは回帰確認で回避できて
おり、モデル強度×カード吸収率の交互作用が見える)。エラーメッセージ品質の差も出た:
Meshは「'eval' is a reserved word and cannot be used as a name」と原因を直接指す一方、
TSは本命のTS1100に加えて`lib.es5.d.ts`内部を指すTS2394(グローバル宣言との衝突の玉突き)
が混ざる。どちらも1往復で復帰したが、原因特定の負荷はMeshの方が低い

## 第2ラウンドの結論

- **改善ループの実証が最大の成果**: 第1Rの発見→修正→回帰確認で往復2→0。
  ベンチが「言語を良くする装置」として機能することを確認した
- モデルを弱くすると差は出るが、その差は**深い機能理解ではなく浅い層**
  (式指向matchの認識・予約語)に留まる。Haikuですらunion・narrowing・spawn/チャネルの
  本体は3言語同等に書けている

  強さも定量で見えた — Meshが目指すべき「浅い層の驚き」の削減目標として参考になる
