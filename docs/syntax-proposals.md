# Mesh 構文候補 — 採用/不採用会 議事資料

> 2026-07-17 起票。各候補は要件定義書 P5 の3基準で評価する:
> ①対訳1行テスト ②例外ゼロ ③家賃を払うか(TS/Goにない価値を生むか)
> 推奨は Claude の意見であり、決定は kanayama さんが行う。

## テーマA: エラー処理

### E1. `!` エラー伝播演算子 — 決定: ◎採用(2026-07-17)

```go
fn getUser(id: int) (User, error) {
	user := fetchUser(id)!        // err != nil なら、そのerrを即return
	profile := fetchProfile(user)!
	return profile.user, nil
}
```

- 対訳: Rust の `?`。Go で毎回書く `if err != nil { return zero, err }` の3行が1文字になる
- 規則: 「最後の戻り値が error の関数の中」でのみ使える。例外なし
- 家賃: AIにも人間にもトークン削減が大きい。エラー処理という本丸への投資
- 備考: 明示的に扱いたい場面では従来の `v, err := f()` も引き続き使える(共存)

### E2. `or` フォールバック — 決定: ◎採用(2026-07-17)

```go
port := parseInt(env("PORT")) or 8080   // エラーなら右の既定値を使う
```

- 対訳: Rust の `unwrap_or` / JS の `??` に近い
- 規則: `(T, error)` を返す式の直後にのみ書ける。例外なし
- 家賃: 「エラーでもいいから既定値で進む」という頻出パターンを1語に

### E3. `must()` 組み込み関数 — 決定: ✕不採用(2026-07-17)

```go
config := must(loadConfig("app.json"))  // エラーなら即panic(停止)
```

- 対訳: Go の must パターン
- 懸念: 「panic = 裏口の例外」を言語の顔にしてしまい P2(暗黙より明示)と緊張関係。
  初期化での用途は E2 や `!` でほぼ代替できる

## テーマB: 並行処理

### C1. `spawn` 式が受取口(task)を返す — 決定: ◎採用(2026-07-17)

```go
task := spawn fetchUser(42)   // 起動と同時に「結果の受取口」を得る
other := doSomething()        // その間に別の仕事
user, err := <-task           // 必要になった時点で待つ
```

- 対訳: JS の「Promise を作っておいて後で await」/ Rust・Erlang の spawn
- 規則: `spawn f()` は式として chan 相当の受取口を返す(文としても使える=戻り値を捨てるだけ)
- 家賃: Go の goroutine(起動が軽い書き味)と JS の Promise(値が返る)の
  文字通りの「いいとこ取り」。Mesh のアイデンティティになり得る独自機能
- キーワードは `go` ではなく **`spawn`**(2026-07-17決定)。Go の `go` は値を
  返さない構文なので、受取口を返す Mesh 版は意味論が別物であり、別名が正しい。
  spawn は Rust/Erlang/Node で並行起動の語として定着しており P5 予算内

### C2. `wait` ブロック(構造化並行) — 決定: ◎採用(2026-07-17)

```go
wait {
	spawn sendEmail(user)
	spawn updateIndex(user)
}
// ここに来た時点で中のタスクは全部完了している
```

- 対訳: `Promise.all` / Go の `sync.WaitGroup` の糖衣
- 規則: ブロック内で起動した goroutine の完了を待ってから抜ける。例外なし
- 家賃: 「起動したまま取りこぼす」バグを構造的に防ぐ(structured concurrency)。
  AIが並行コードを書くときの安全柵になる

### C3. `select` 文(Goのまま導入) — 決定: ◎採用(2026-07-17)

```go
select {
case msg := <-ch1:
	print(msg)
case <-done:
	return
}
```

- 複数チャネルの「先に来た方」を待つ。タイムアウト・キャンセル処理の要
- 対訳: Go の select そのまま
- 推奨は「後回し」だったが採用と決定。実装順は wait/spawn より後でよい
  (default節・タイムアウトの標準形など、詳細設計は言語仕様書で行う)

## テーマC: 型と関数

### T1. struct 宣言の構文 — 決定: 案c採用(2026-07-17)

```go
// 案a: Go風
type User struct {
	name: string
}

// 案b: TS風
type User = {
	name: string
}

// 案c: Mesh風 — 推奨: ◎これ
type User {
	name: string
	age: int
}
```

- 案cの対訳: TS の interface 相当。`struct` や `=` という冗長な記号を削った最短形
- 規則: フィールドは `名前: 型`(関数引数と同じ形)。例外なし

### T2. union 型(TSの最強機能) — 決定: ◎採用(2026-07-17、段階的に実装)

```go
type Status = "active" | "banned" | "pending"

fn ban(s: Status) { ... }
ban("actev")   // ← コンパイルエラー。AIのタイポ・幻覚を型で捕まえる
```

- 対訳: TS のユニオン型そのまま
- 家賃: Go に無い価値の代表。AIの間違いをコンパイル時に捕まえる網が太くなる
- 実装は段階的に: まず文字列リテラル union → その後一般の union

### T3. 文字列補間 — 決定: ◎採用・A方式(2026-07-17)

```go
msg := "worker ${id} done"   // "worker " + str(id) + " done" が不要に
```

- 対訳: TS のテンプレートリテラル(バッククォートではなく通常の `"` で使える)。Kotlin と同方式
- 規則: `${式}` は常に補間。リテラルの `$` が必要なら `\$`。例外はこのエスケープ1つ
- 検討した代替案: B) バッククォート(TS完全互換)は文字列が2種類になりP1に反し、
  「`"` 内の `${}` は展開されない」という静かな罠を輸入する。C) Python流 f-string は
  f を忘れてもエラーにならず静かに壊れる。D) Swift `\()` は学習データ僅少。
  E) 補間なし(Go流)は str() 連結が冗長。
  **A は「間違いが必ず構文エラーとして大声で壊れる」唯一の案**(P4)のため採用

## 確認事項(現状維持の追認)

- 関数キーワードは `fn`(トークン最短。Rust/多数の新言語と同じ)
- 宣言は `:=`、条件の丸括弧なし、セミコロンなし(v0のまま)
