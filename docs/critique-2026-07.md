# 言語批評ターン(2026-07-19)

> モジュールシステム実装完了時点でのMesh言語設計の批評記録。
> 手法: ①言語設計者レンズ・②実務/AIエージェントレンズの2体のサブエージェント
> (設計経緯の先入観なし・**実際にMeshコードを書いて実機検証**)+③設計経緯を知る
> Claude本体の内部批評、の3視点を統合した。
> ここから出た再討議項目は [design-agenda.md](design-agenda.md) の **F節** で追跡する。

## 最重要: 2体の独立批評が一致した5点

独立した批評が同じ場所を突いた項目(信頼度が高い):

| # | 一致した指摘 | 設計者レンズ | 実務レンズ |
|---|---|---|---|
| 1 | **関数型注釈が無く、ユーザーが高階関数を書けない** | 「穴ではなく崖。Go 1.0の失敗の再演、しかもGoより悪い」 | 「retry(f)・sortByが全滅。抽象化の天井が極端に低い」 |
| 2 | **エラーが文字列のみで、`!`伝播と構造化エラーが排他** | 「背骨(union)と看板(`!`/`or`)が噛み合っていない」 | 「`!`で伝播すると行番号等のコンテキストが消失」 |
| 3 | **フィールドのnarrowingが黙って効かない** | 「カード推奨の再帰structパターンが自傷している」 | 「TSの感覚でAIが確実に書き、黙って失敗する」 |
| 4 | **本当の独自資産は型システムではなくツールチェーン契約** | 「カード+JSON診断+CI突き合わせは他言語に無い発明」 | 「一発成功率5/6はカードの出来による。診断に機械適用可能なfixを」 |
| 5 | **struct更新構文(spread相当)が無い** | 「全フィールド再列挙は構造的型付けと複合して危険」 | 「フィールド8個で1個変えるのに8行」 |

## 統合サマリ

### 良いところ(3視点の総意)
- **言語カード+`check --json`+CI突き合わせ**は実体がある(実測: 新言語なのに一発コンパイル成功率5/6)
- union一本の規律(none/error/closed/map読みが全部同じnarrowing)
- 2段所有の構造化並行(「Goに対する本物の改善」)
- 判別可能union+網羅的match、`m[k] or 0`、名前空間分離、デフォルト不変

### 悪いところ(重要度順)
1. 高階関数の崖(上表#1)。`filter/transform/reduce`だけコンパイラ特権という不平等も
2. エラー設計の分裂(上表#2)。実アプリがエラー種別分岐を要した瞬間、stringly-typedに退行するか`!`を捨てるかの二択
3. **入出力が一切ない**(stdin/argv/file/HTTP/JSON/日付/乱数ゼロ)。実務レンズの一発成功はすべて自己完結計算だったから、という自己分析つき
4. **構造的型付けの全面採用はP2と衝突**。`struct Meters`の値が`Dollars`を取る関数にそのまま渡る実証あり(単位型・ID型の事故)
5. **int 53bit問題**: `9007199254740992 + 1`が+1を無視する実証。panic哲学と矛盾する意味論漏れ
6. **narrowingの弱さがTS学習データと負の転移**: `a is none || b is none`・`!(a is none)`・`a is int && a > 3`・フィールドパスが全滅

### イケていないところ(抜粋)
- `transform`という名前(「妥協の経緯ごと公開されている敗北」。文脈依存キーワード化の再検討提案)
- `!`は記号選定ミスの疑い(後置`!`の学習データはSwift/Kotlin=crash、TS=no-op。「伝播」はRustの`?`。負の転移)
- 空の型付き配列が2通り(`Todo[]{}` / `xs: T[] = []`)— P1を公式仕様が自分で破っている
- `or`がerrorまで無音で吸う(失敗の事実がログにも残らない最短経路)
- 判別可能unionのメンバー推論は遠隔作用(フィールド集合で解決先が決まり、メンバー追加で離れたリテラルが壊れる)。
  内部者の対案: タグ(`kind`)必須を課せばほぼ消える
- 全関数async化の意味論(全呼び出しがyield point)がカード未記載。性能は素のJSの約3倍遅(fib実測)
- 構文エラーが1件で停止(型エラーは一括なのに)
- カードの完全性の穴: `s[0]`が動くのに未記載 / `toFloat`・`round`等の数値系欠落 /
  selectの擬似ランダム選択は「決定性」の看板と実は矛盾

### 改善提案(統合・優先度順)
1. **関数型注釈**(`fn(int) int`)— 両批評の最重要。単相なら実装も小さい
2. **`!`の一般化** — 「成功型を除いた残りが呼び出し元の戻り値型に代入可能なら伝播」+ `toInt(s) ! "line ${i}"`のコンテキスト付加形
3. **narrowing拡張** — 最低でもフィールドパスと`&&`連結。「効かないパターン」のカード明記は即日可能
4. **`mesh/io` + `mesh/json`** — JSONは自己参照判別可能unionのショーケースになる
5. **最小ジェネリクス** — 成功判定は「contains/indexOfを組み込み特権からstdlib通常関数に降格できること」
6. struct更新構文、`get(arr, i) → T | none`、数値stdlib、`mesh test --json`、構文エラー復帰

### 独自性(3視点の結論)
- **賭けるべきは「ツールチェーン契約」**: 診断にエラーコード+機械適用可能な`fix`パッチ、`mesh explain`、
  サブセットカード。「仕様・診断・修正候補がテストで同期される言語」は世界に存在しない
- **「TypeScriptでよくない?」への答えは3点に絞る**: ①リークが構文的に書けない並行、
  ②不在/closedを無視できない強制、③抜け穴の無さ(→`any`の存在は③を毀損)
- **最大のリスク: 核心仮説が未測定**。対TS/Go比較ベンチ(一発成功率・修正往復数・diff一貫性)を
  一度もやっていない。カード実験第1〜10回は「穴探し」であって「比較測定」ではない
- **P6は現状空手形**: 型共有だけならTSが既にやっている。「API境界で型が検証される」まで行って初めて存在理由

### 総括
コア(union・判別可能union・構造化並行)とツールチェーン思想は本物。だが「抽象を作れない」
「エラーに文脈を載せられない」「TSの直感が通らない」の3つの崖は、いずれも**この言語自身の
核心仮説(AIの正答率)を下げる方向に働く**点で、機能不足より深刻。
推奨順: ①関数型注釈 → ②エラー設計の和解 → ③対TS/Go測定。

## 付録A: 言語設計者レンズの批評(全文)

検証はすべて実機で行った。以下、各指摘は「主張 → 根拠 → 対案」の形式。

### A-1. 良いところ

**1-1. 言語カード+`check --json`+CI突き合わせは、本物の独自資産である。**
「AIが書く」を掲げる言語は他にもあるが、仕様書を「モデルのコンテキストに収まる1枚」として設計し、カードの主張とコンパイラ実装の乖離をCIで落とす仕組みまで持つ言語は存在しない。P4はマーケティングではなく実体がある。特に「存在しない機能リスト(Does NOT exist)」と「頻出エラー→直し方」の対をカードに含める発想は、ネガティブ仕様の明文化としてAI向け言語設計の発明と言っていい。

**1-2. union一本での不在・失敗・終端の統一は一貫している。**
`T | none`(不在)、`T | error`(失敗)、`V | none`(map読み)、`T | closed`(チャネル受信)がすべて「narrowingするまで使えない」という単一の規律に従う。TSのnull/undefined二重空値も、Goのcomma-okとzero-valueの罠も輸入していない。`== none` を禁止して `is none` に一本化したのも正しい。

**1-3. 2段所有の構造化並行(spawn/detach)は6原則に最も忠実な発明。**
「関数所有=暗黙wait / プログラム所有=detach」の2値は、Goが context・errgroup・goroutine leak detector で事後対処している問題を構文で殺している。`detach` が `mut` と同様にgrep可能な「レビューの注視点」になる設計はP2/P3と綺麗に整合する。新規性予算(P5)の使い所として正しい。

**1-4. デフォルト不変+`mut`+シャドーイング禁止+グローバル変数なしの組み合わせ。**
「一つの名前=一つの値」が関数ローカルで保証され、生成JSも素直に const/let に落ちる。P3の実装として堅い。

**1-5. メソッドと自由関数の名前空間完全分離。**
`render(t)` と `t.render()` の両方が通る言語(UFCS)はAI出力を割る。「宣言ごとに呼び方は常に1通り」は地味だがP1の最良の適用例。

**1-6. mapの挿入順保証と決定性志向。**
Goのmap順序ランダム化と逆張りだが、「同じ入力→同じ出力」と診断の再現性に効く。AIの検証ループ前提なら正解。

### A-2. 悪いところ

**2-1. ユーザー定義の高階関数が構文的に不可能。これは穴ではなく崖である。**
実測: `fn apply(f: fn(int) int, x: int) int` は `expected 'ident' as type name, but got 'fn'` でパースエラー。関数型注釈が存在しないため、関数を引数に取る関数がstdlib以外に書けない。一方で `filter/transform/reduce` はコンパイラ組み込みとして特権的に多相。これはGo 1.0の「stdlibだけジェネリック(append/make)」の失敗の完全な再演で、Goは撤回に10年かけた。しかもMeshはGoより悪い——Goはinterface{}という逃げ道があったが、Meshには関数を受け取る手段そのものがない。抽象を作る道具を与えずに手書きを強いるのは、AIに毎回同じボイラープレートを生成させ、P1の「出力の一貫性」成功基準を自分で毀損する。
対案: 最低限、関数型注釈を先に入れる。ジェネリクスはA-4-1参照。

**2-2. 構造的型付けの全面採用はP2「暗黙より明示」と正面衝突している。**
実測: `struct Meters { value: float }` の値が `Dollars` を取る関数にそのまま渡り、`paid 100` が出力された。単一フィールドstructはドメインの単位型・ID型(UserId vs OrderId)として最頻出のパターンであり、まさにそこが最も静かに事故る。TSが構造的なのはJSの漸進型付けという歴史的制約のためだが、Meshはグリーンフィールドなのにその制約だけ輸入した。GoもRustも名前的で、構造的型付けの「意図しない互換」はTSコミュニティ自身が branded types というハックで戦っている問題。「JSON/API境界と透過的」という動機は境界だけ構造的にすれば足りる。
対案: デフォルト名前的+境界(JSON decode / 判別可能unionの無名メンバー)のみ構造的。または opt-in の `distinct struct UserId { value: string }`。

**2-3. `int` が53bit JS numberのまま、静かに壊れる。**
実測: `9007199254740992 + 1` が `9007199254740992` を返す(+1が消える)。範囲外アクセスとゼロ除算には位置つきpanicを実装して「JSの意味論を漏らさない」と宣言しながら、最も基本の型で最も検出しにくい形の意味論漏れが残っている。ID採番・タイムスタンプ演算・金額計算で実害が出る型の嘘。
対案: 整数演算結果の `Number.isSafeInteger` 検査をpanic層に追加(「バグは即停止」思想の適用)。

**2-4. narrowingが「文頭の `if x is ...` 単独形」しか効かず、TSの学習データと最悪の形で乖離している。**
実測で全滅したもの: `!(a is none)` / `a is none || b is none` / `a is int && a > 3` / そして**フィールドアクセス `if n.next is none { ... } sum(n.next)`**。最後のは深刻で、カード自身が推奨する再帰structのcanonicalパターン(`next: Node | none`)が、いちいちローカル変数へコピーしないと辿れない。「TypeScriptの型」を看板にする言語で、TSのnarrowingの最頻イディオムが軒並みコンパイルエラーになるのは、核心仮説「言語設計がAIの正答率を左右する」に照らすと、正答率を積極的に下げる設計になっている。

**2-5. `error` がメッセージ文字列のみの不透明型で、構造化エラーと `!`/`or` が二者択一になっている。**
実測: 失敗を判別可能union(`type DbError = { kind: "notFound", table: string } | ...`)で表現した瞬間、`'!' has nothing to propagate`、`left side of 'or' never fails` となり、言語の看板機能が全部使えなくなる。実用アプリがエラーの種類分岐(リトライすべきか、404を返すべきか)を必要とした瞬間、stringly-typed エラーに退行するか、伝播の糖衣を全部手書きmatchに戻すかの選択を迫られる。Goが `errors.Is/As` を後付けで発明した理由、Rustが `?` を `From` 変換つきで設計した理由がここにある。言語の背骨(union)と看板(`!`/`or`)が噛み合っていない。

### A-3. イケていないところ

**3-1. `transform` という名前は、妥協の経緯ごと公開されている敗北である。**
学習データ上 `map` の優位は圧倒的で、「AIが必ず書き間違える名前を選び、エラーメッセージで矯正する」のはP4でP5の失敗を尻拭いする構図。型位置の `map<` と式位置の `map(` は字句的に区別可能なので、文脈依存キーワード化で両立できたはず。「頻出エラー欄に載る名前」の方が一貫性のコストとして大きい。

**3-2. 空の型付き配列の書き方が2通りあり、カード自身が「same result, more familiar」と併記している。**
`Todo[]{}` と `xs: Todo[] = []`。P1を公式仕様書が自ら破っている数少ない箇所で、しかも新参者(AI)が最初に触る構文。どちらかを削るべき。

**3-3. 配列の範囲外はpanic、mapの欠損キーは `V | none`——不在哲学の非一貫。**
`xs[i]` の範囲外が常にバグとは限らない(ユーザー入力由来のindex等)。mapには型で強制される安全な読みがあるのに、配列には `len` で手動ガードする以外の道がない。
対案: `get(arr, i) T | none` を組み込みに追加(`arr[i]` はバグ検出用として残す)。

**3-4. 伝播記号 `!` は「3人の偽の友達」を持つ最悪の選定。**
学習データで後置 `!` は Swift(force unwrap=crash)、Kotlin `!!`(crash)、TS(non-null assertion=実行時no-op)であり、どれも「呼び出し元へ伝播」ではない。意味が完全一致する Rust の `?` が既にあるのに、それを捨てて意味の異なる既存記号を再利用したのは、新規性予算の誤用——記号が既知でも**意味が違えば負の転移**になる。前置 `!`(否定)との同居も認知負荷を上乗せする。

**3-5. 判別可能unionの「union名+フィールド集合からのメンバー推論」は進化ハザードを抱えた遠隔作用。**
フィールド集合が重なるメンバーも宣言でき、リテラルの解決先は「他のメンバーに何があるか」で決まる。メンバーやフィールドを1つ追加すると、離れた場所の既存リテラルが曖昧エラー化するか解決先が変わる。これはP3「局所で読める」に反する。Rustのvariant名(`Shape::Rect{..}`)やTSのタグ明示が避けた道。

**3-6. 全関数async・全呼び出しawaitにより、原子性の単位が事実上ゼロになっている。**
生成コード確認: すべての関数呼び出しがyield pointになるため、spawnされたタスクと共有する `mut` 状態・配列・mapは、どの2つの呼び出しの間でも割り込まれうる。「シングルスレッドだから安全」に見えて、同期機構は無く、race detector相当の検出もない。カードはこのinterleaving意味論に一言も触れていない。性能は fib(27) で素のJSの約3倍(45ms vs 13ms)。

**3-7. `or` は none と error を同一視して失敗を無音で握りつぶす。**
`port := parseInt(s) or 8080` は「エラーが発生した事実がログにすら残らない」への最短経路で、AIは喜んで多用する。Swiftの `try?` が批判され続けているのと同じ構図。

**3-8. グローバル定数が無いため、設定値・定数表の置き場がない。**
`fn maxRetries() int { return 3 }` のような関数化を強いるが、これはP1が嫌う「同じことの別の書き方」を各プロジェクトが発明する余地になる。不変なら共有可変状態の問題は生じないのだから、トップレベル不変束縛は禁欲しすぎ。

### A-4. 改善提案

**4-1. 関数型注釈→最小ジェネリクスの順で、抽象化の崖を埋める。**
```go
fn apply(f: fn(int) int, x: int) int { return f(x) }      // まず関数型注釈

fn findFirst<T>(arr: T[], pred: fn(T) bool) T | none {    // 次に fn 限定のジェネリクス
    for _, v := range arr {
        if pred(v) { return v }
    }
    return none
}
```
これが入れば `contains/indexOf/filter/transform/reduce` は組み込み特権をやめてstdlibの通常関数に降格でき、言語の一貫性が上がる。

**4-2. narrowingを「安定パス+論理演算子」へ拡張する。**
最低ラインは (a) `&&` の左辺の `is` が右辺とthen節に効く、(b) `!` と `||` のド・モルガン、(c) フィールドの安定パス。実装コストが問題なら、せめて (c) だけでも先行させるべき。

**4-3. `!` を「戻り値型との差分をすべて伝播」に一般化し、構造化エラーと和解させる。**
```go
type DbError = { kind: "notFound", table: string } | { kind: "timeout", ms: int }

fn useIt(id: int) int | DbError {
    v := find(id)!        // findの失敗メンバー(DbError)が呼び出し元の戻り値型に
    return v + 1          // 含まれるなら伝播できる、と再定義する
}
```
「none/error 限定」より「成功型を除いた残りが呼び出し元の戻り値型に代入可能なら伝播」の方が規則として単純で例外がない。

**4-4. struct更新構文を入れる。**
```go
fn (t: Todo) complete() Todo {
    return Todo{...t, done: true}      // 現状は title を毎回書き写す
}
```

**4-5. 伝播記号を `?` にする(今なら間に合う)。** v1前の今が最後の改名機会。

**4-6. `or` を none 専用にし、error には理由へのアクセスを提供する。**
```go
n := toInt(s) or 0                  // error まで無音で吸うのはやめ、
n := toInt(s) or e { log(e); 0 }    // 捕まえた失敗を受け取れる形を追加する
```

**4-7. 配列に型安全読み `get(xs, i) → T | none` を追加する。**

**4-8. match の subject 束縛(`match r := getUser(id) { ... }`)を許す。**

### A-5. 独自性

**5-1. 最大の差別化資産は型システムではなくツールチェーン契約——ここに投資を集中すべき。**
union/narrowing/structural typingは要素技術としてはTS/Rust/Gleamの既知集合であり、「寄せ集め」批判に対する防御力はない。他方「カード=検証済み圧縮仕様」「--json診断」「エラーメッセージに修正方法を含める規約」は他言語が持たない。診断JSONに機械適用可能な `fix`(置換パッチ)フィールド、`mesh explain <code>`、使用機能だけのサブセットカード等、「コンパイラがAIの相棒」を診断文言のレベルからプロトコルのレベルに上げれば、これはMeshにしかない価値になる。

**5-2. 2段所有の構造化並行は看板にできるが、「無制限バッファ既定」の説明責任が残っている。**
既定 `chan<T>()` = 無限バッファはGoが意図的に拒否した設計(バックプレッシャー喪失)。detachされた常駐ワーカー+無限バッファは「goroutineリークは死んだがメッセージリークは生きている」という新しい形の資源リークを許す。仕様として自覚的に文書化するか、既定を有限に再検討すべき。

**5-3. 「AIの正答率を上げる」という核心仮説が、いまだに測定されていないのが最大のリスク。**
requirements 7 に成功基準(一発成功率・修正ループ数・出力一貫性)があり、カード実験で穴出しはしているが、**対TS/Go比較のベンチマークは未実施**。測定なしの「AI-first」は、他言語が「開発者体験ファースト」を名乗るのと同程度のマーケティングに転落する。TODOアプリ仕様でMesh/TS/Goの3言語比較を回すのが、次にやるべき最も価値のある一手。

**5-4. P6「フルスタック一体」は現時点で空手形であり、TSに対する存在理由が宙に浮いている。**
「1つの型定義をフロント/バックで共有」はTSが既に実現している価値。「共有した型がAPI境界で*検証される*」(型付きRPC、JS境界でのnull/例外正規化)まで進んで初めてMeshが選ばれる理由が生まれる。境界正規化を言語仕様として持つ計画は、TSのZod的な後付け検証を言語に内蔵する話であり、独自性の種として有望。

## 付録B: 実務/AIエージェントレンズの批評(全文)

### B-0. 実録: 実際に書いた5本+プローブで何が起きたか

- **inventory.mesh** — 在庫管理(struct配列+判別可能union `OrderResult` 3種+match)。**一発コンパイル・一発正常動作**。
- **sales.mesh** — CSV風文字列の部門別集計+売上降順表示。一発で通ったが、**「structを値でソート」ができず選択ソートを25行手書き**。また `!` で伝播したエラーが `skip: "x99" is not a valid int` と表示され、**何行目のどのフィールドかというコンテキストが完全に消失**。
- **retry.mesh** — リトライ+バックオフ。一発動作。ただし `mut lastErr := error("unreachable")` という**番兵エラーのダミー初期化**を書かされた。
- **hof.mesh** — 汎用 `retry(f)` を書こうとして**撃沈**: 関数を引数に取るユーザー定義関数は書けない。
- **pool.mesh** — 並行フェッチ+`wait`+`close`+`closed` narrowing+集約。**一発動作**。書き味は今回の中で一番良かった。
- プローブ結果: 型エラーは3件まとめて報告されるが構文エラーは1件で停止。`main` 内の `!` は「戻り値型に error を足せ」と提案されるが**mainは戻り値型を持てないので実行不能なアドバイス**。`if r.body is error { return }` は**通るのに何もnarrowしない**。`s[0]` は**カード未記載なのに動く**。`toFloat` は存在しない。`0.1+0.2` は `0.30000000000000004` と出力され、**丸める手段が言語内に無い**。

一発成功率は新言語として異常に高い(5/6)。カード+check --json の効果は本物。ただし成功したのは全て「入出力のない自己完結計算」であることが、以下の批評の核心。

### B-1. 良いところ

**1-1. カード→生成→`check --json` のループは実際に回り、一発成功率が高い。**
「Does NOT exist」リストと頻出エラー表が、AIが書きがちなJS/Go混入(`arr.map`、`v, ok :=`、`== none`)を事前に潰している。

**1-2. 判別可能union+網羅的matchは「業務のエラー分岐」を実際に型で守る。**
`OrderResult`(ok/outOfStock/unknownSku)で、`res.total` は `{kind:"ok"}` アーム内でしか触れず、アームを1本消すと「missing: ...」で名指しされる。**文字列エラーの唯一の実用的な代替がここにあり、ちゃんと機能する**。

**1-3. `m[k] or 0` の集計イディオムは簡潔かつ安全。**
CSV集計では一度もキー存在バグを踏まなかった。

**1-4. 構造化並行(spawn所有権+`T | closed`)は看板に偽りなし。**
受信が常に `T | closed` なのは、Goで頻出する「closeし忘れ/受信ループが終わらない」バグを型で殺しており、**Goに対する本物の改善**。

**1-5. 型エラーの一括報告+修正方法つきメッセージは修正ループの回転数を下げる。**

**1-6. デフォルト不変+シャドーイング禁止は「AIの書き癖」に効いている。**
「mut と detach だけgrepすればいい」というレビュー導線は実務者に刺さる。

### B-2. 悪いところ(致命的)

**2-1. 入出力が一切ない。stdin・argv・環境変数・ファイル・HTTP・JSON・日付・乱数、全部無い。これが最痛。**
組み込みの副作用は `print` と `sleep` のみ。**Meshプログラムは外界から1バイトも受け取れない**。最低限の `mesh/io`(args/stdin/readFile)と `mesh/json` が来ないと、言語の他のどの改善も検証できない(実データを食わせられないから)。

**2-2. ユーザー定義の高階関数が書けない=抽象化の天井が極端に低い。**
`retry(f, 3)`・`sortBy(items, key)`・ミドルウェア・イベントハンドラといった**業務コードの基本語彙が全滅**。

**2-3. `error` が文字列しか持てず、`!` 伝播でコンテキストが消える。実務のエラーハンドリングに耐えない。**
Goなら `fmt.Errorf("line %d: %w", i, err)`、Rustなら `.context()` に当たるものが無く、コンテキストを足すには毎回3行書く必要があり、それをすると `!` の存在価値が消える。**「楽な伝播」と「情報のあるエラー」が二者択一**になっているのが構造的欠陥。

**2-4. ジェネリクス不在の実害は「ライブラリを書けない」という形で既に出ている。**
モジュールシステムを作った今、「パッケージとして共有できる汎用コードが型的に書けない」ため、エコシステムが原理的に立ち上がらない。

**2-5. `sort` にコンパレータが無く、structの配列を並べ替えられない。**
集計→ランキング表示は業務コードの最頻出パターンであり、**初日に全員が踏む**。

**2-6. 数値まわりの道具が皆無: `toFloat`・`abs`/`min`/`max`/`floor`/`round`・桁数フォーマットが無い。**
金額・比率・パーセントを扱う業務コードで即死。

### B-3. イケていないところ(摩擦)

**3-1. フィールドのnarrowingが「黙って効かない」。カードにも書いていない。**
カードは「COMPLETE reference」を名乗る以上、**効かないnarrowingパターンこそ明記すべき**(AIはTSの感覚で確実にこれを書く)。

**3-2. `!` がmainで使えず、しかも診断が実行不能な修正を提案する。**
エントリポイントでは結局全部手動narrowingになり、スクリプト的な用途で一番書きたい場所で一番冗長になる。

**3-3. struct更新にspread相当が無く、全フィールド書き写し。**

**3-4. 配列操作が push だけ。remove/pop/slice/concat が無く、「除去」は毎回配列再構築。**

**3-5. タプルが無いので、その場限りのペアに毎回名前つきstructか文字列配列を強いられる。**

**3-6. 番兵エラーのダミー初期化(`mut lastErr := error("unreachable")`)という不格好なイディオムが必要。**
「ループで最後の失敗を覚える」という頻出パターンに定型が無い。

**3-7. 構文エラーは1件で停止し、AIループの回転数を食う。**
型エラーは一括なのに構文だけ逐次で、**1構文エラー=1往復**のコストになる。

**3-8. カードの「完全性」に既に穴がある: `s[0]` は動くが未記載。**
「カードに無い機能は存在しない」という契約が売りなのだから、**実装がカードからはみ出すこともCIで殺すべき**。

**3-9. `+=` が無いのに `++` はある、という一貫性の欠け方。**
`n = n + 1` と `n++` の2通りを既に許している以上、「書き方を一つに」では `+=` 拒否を説明できない。

### B-4. 改善提案(優先度順)

**P0-a. `mesh/io` と `mesh/json` を最優先で。JSONはせっかくの自己参照DUの見せ場。**
```go
import "mesh/io"
import "mesh/json"

fn main() {
    text := io.readFile(io.args()[0])!   // string | error
    v := json.parse(text)!               // json.Value | error
}
// json.Value は自作機能のショーケースそのもの:
// type Value = { kind: "str", s: string } | { kind: "num", n: float }
//            | { kind: "arr", items: Value[] } | { kind: "obj", entries: map<string, Value> }
//            | { kind: "bool", b: bool } | { kind: "null" }
```

**P0-b. 関数型注釈の解禁(まず単相だけでも)。**

**P1-a. エラーへのコンテキスト付加を1トークンで:**
```go
amount := toInt(trim(parts[2])) ! "line ${i}: bad amount"
// 失敗時 "line 4: bad amount: \"x99\" is not a valid int" として伝播
```

**P1-b. ジェネリクス(Go 1.18程度の最小構成)。**
**P1-c. `sortBy` / コンパレータ(P0-bが前提)。**
**P2-a. 数値stdlib: `toFloat` / `abs` / `min` / `max` / `floor` / `round` / `formatFloat(x, 2)`、乱数。**
**P2-b. struct更新構文。**
**P2-c. 構文エラーのリカバリつき複数報告。**
**P2-d. `mesh test` を `--json` 出力つきで。**「型は通ったが挙動が違う」を機械可読で返せて初めて生成→検証ループが完結する。
**P3. フィールドnarrowingの実装、または「効かない」ことのカード明記+誘導文。**

### B-5. 独自性

**5-1. 本当のモートは「CI検証済み言語カード+機械可読診断」という契約。ここを製品化しきるべき。**
診断にエラーコードと機械可読な `fix` フィールド(`{"code": "E-IMMUT", "fix": {"insert": "mut ", "at": ...}}`)を足せば、エージェントはメッセージ解釈すら不要になる。

**5-2. 「TypeScriptでよくない?」への現状の答えが弱い。TSが原理的にできない3点に賭けるべき。**
(1) spawn所有権による「リークが構文的に書けない」並行、(2) map読み/チャネル受信の「不在・closedを無視できない」強制、(3) 抜け穴の無さ(TSは `as any` で全てが崩れる)。READMEの顔をunionではなくこの3点にすべき。Meshの `any` は第3の主張を毀損している。

**5-3. 「AIが書き、人間がレビューする」なら、レビュー可能性を言語機能に昇格させると唯一無二になる。**
副作用・失敗しうる呼び出しが呼び出し側の字面に必ず現れる(IO関数は `io.` 修飾必須+失敗は `!`/`or`/`is` のどれかが必ず可視)ところまで行くと、「diffを読むだけでリスク箇所が全列挙できる言語」という、TSにもGoにも無い売りになる。stdlib設計をこの原則で貫くのが分水嶺。

**5-4. 乗り換え動機は現状「人間には無い」。ターゲットを「エージェントに書かせる運用者」に絞るべき。**
TS/Go経験者の学習コストは実測でほぼゼロだった。「人間が学ぶ言語」ではなく「人間はカードとdiffだけ読めばいい言語」として売る方が実態に合う。ただしIOが塞がってからの話。

### B-総括
コア(union/narrowing/DU/構造化並行)は新言語として驚くほど完成度が高く、カード+checkのループも宣伝通り機能する。しかし現状は「入出力を持たない、抽象化のできない、エラーに文脈を足せない言語」であり、優先順位は明確で、**(1) IO/JSON、(2) 関数型注釈、(3) エラーコンテキスト、(4) ジェネリクス**。この4つが入るまでは「よくできたプレイグラウンド」の域を出ない。
