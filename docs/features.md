# Mesh 機能一覧(できる・できない表)

> Mesh の機能の**現在地**を一目で見るための表。新しい決定はこの表を更新する。
> 経緯・理由は [syntax-proposals.md](syntax-proposals.md) と [requirements.md](requirements.md) を参照。
>
> - ✅ **できる** — v0 で実装済み、今すぐ動く
> - 🔜 **できる(実装待ち)** — 採用決定済み、実装はこれから
> - ❌ **できない(意図的)** — 設計判断としてやらない。理由つき
> - ❓ **未決** — まだ議論していない

## 値と型

| 機能 | 状態 | 備考 |
|---|---|---|
| 静的型付け+型推論 | ✅ | `x := 10` で右辺から推論 |
| プリミティブ型 int / float / string / bool | ✅ | int同士の除算は切り捨て |
| 配列 `T[]` | ✅ | `nums := [1, 2, 3]`。空の型付き配列は `Todo[]{}`(素の `[]` は `any[]`)。多次元 `int[][]{}` も可。
      2026-07-19: 総称型を要素にする配列型 `chan<T>[]` / `map<K,V>[]` に対応(カード実験第7回)。
      複数行の配列リテラルは末尾カンマ無しでもOK(struct/mapリテラルと同じASI対応。同第8回) |
| struct 定義 | ✅ | 2026-07-17実装。`struct User { name: string  age: int }`。
      生成は `User{name: "alice", age: 30}`(全フィールド必須・不明/重複/型不一致は検出)。
      フィールドアクセス `u.name` は本物の型検査つき(**タイポは fields一覧つきエラー**)。
      再帰struct(`next: Node | none`)対応。if/for/matchヘッダでの裸のリテラルは禁止(Goと同じ)。
      同一性は**名前的**(2026-07-19 F-3決定、下の行を参照) |
| 型の同一性: **名前的**(無名のみ構造的) | ✅ | 2026-07-19 F-3決定(批評起点。全面構造的を同日中に巻き戻し)。
      **名前付きstruct同士は名前で判定** — 形が同じでも `Meters` と `Dollars` は別の型で、
      取り違えは `cannot use Meters as Dollars` のコンパイルエラーになる(単位型・ID型の
      「静かな事故」を防ぐ。Go/Rustと同方式)。**無名`{...}`(判別可能unionメンバー)が絡む
      比較だけ構造的**(名前が無いので形で比べるしかない)— これにより「名前付きstructを
      無名メンバーの場所に渡す」互換は維持される。再帰struct比較の無限再帰ガードつき |
| インライン型式 `{ kind: "ok", ... }`(判別可能union) | ✅ | 2026-07-19実装(C-1)。`type` 宣言の
      union内だけで書ける無名struct型。`type X = { ... }` と裸で書くのは今まで通りエラーで
      「struct X を使え」と誘導(書き方は常に1通り、B-5)。値は **union自身の名前**を struct
      リテラル名として使う(`GetUserResponse{kind: "ok", user: u}`) — フィールド集合(曖昧なら
      さらに値の型)から該当メンバーを特定。match は部分構造パターン `{ kind: "ok" }` で
      絞り込む(フィールド束縛はなし、絞り込み後は普通のフィールドアクセス)。
      **自己参照(木構造・ASTなど)は2026-07-19実装済み** — structフィールド越しの参照
      (`{ kind: "node", left: Tree, right: Tree }`)なら知恵の輪(knot-tying)で解決できる。
      structを挟まない「union同士が裸で直接参照し合う」形(`type A = B \| none` かつ
      `type B = A \| error`)だけは引き続き `type alias cycle` エラー(flatten時に相手の
      placeholderがまだ空で型情報が消える不具合を避けるため、意図的に据え置き) |
| union 型(**言語の背骨**) | ✅ | 2026-07-17実装。`int \| error`、`string \| none` などの union が型注釈に書ける |
| 文字列リテラル型 | ✅ | 2026-07-17実装。`"active"` は string の部分型。リテラルは不変宣言でリテラル型のまま、
      mut 宣言・配列要素では string に widening。**タイポは `cannot use "actev" as ...` で即検出** |
| `type` 宣言 | ✅ | 2026-07-17実装。`type Status = "active" \| "banned"` / `type Result = int \| error`。
      循環・重複・組み込み名の再宣言は検出。右辺に裸の `{...}` は「structを使え」エラー(B-5) |
| `T \| none` による不在表現+narrowing | ✅ | 2026-07-17実装。`if x is none { return }` で型が絞り込まれる。
      then内・else内・(thenが必ず抜けるなら)後続行にも効く。
      **絞り込み前に中身に触るとコンパイルエラー**(未処理の不在が機械的に弾かれる)。
      2026-07-19: `is` のパターンを **match と完全に同じ**へ拡大 — none/error/closed に加えて
      型名(`u is User`)・文字列リテラル(`s is "active"`)・部分構造(`res is { kind: "ok" }`)。
      部分構造は複数メンバーに一致しうるので then側=一致したメンバーのunion / else側=残りのunion。
      2026-07-19(F-6): narrowingを**フィールドパス・`&&`・`!`/`||`**へ拡張。
      `if n.next is none { ... }` はローカル変数へコピーせず直接絞り込める(rootが不変変数な
      ら任意の深さ)。`a is Foo && a.value > 0` は左の`is`が右辺の式とthen節の両方に効く。
      `!(a is none)` / `a is none \|\| b is none { return }` はド・モルガンで絞り込まれる
      (else側=両方不成立の積)。フィールドへの代入は古い絞り込みを無効化してから型検査する
      ので健全性は保つ。`match`の対象もフィールドパスまで絞り込めるよう同じ仕組みに統一した |
| 汎用の空値(null / nil) | ❌ | 2026-07-17決定(union路線)。「どこにでも入り得る空値」は存在しない。
      不在は `T \| none` と書いた場所にだけ、型として現れる |
| `any` 型 | ✅ | 脱出口。乱用しない |
| map 型 | ✅ | 2026-07-18実装。`map<string, int>{"a": 1}`(chan<T>と同系の記法)。
      読み取り `m[k]` は **`V \| none`** を返す(無いキーは or / is none / match で処理するしかない)。
      書き込み `m[k] = v`(値の型で検査)、`delete(m, k)`、`len(m)`。実行時はJSのMap |
| ジェネリクス | ❓ | 未議論(Goも後から入れた機能。急がない) |
| クラス・継承 | ❌ | P1: 書き方を一つに。データはstruct、振る舞いは関数 |
| enum | ❌ | union型で代替 |
| `undefined` | ❌ | 不在は `none`(union内)だけ。JSの2空値問題(null/undefined)を輸入しない。
      JS境界では null / undefined のどちらも `none` に正規化する予定(下のモジュール・相互運用を参照) |
| 暗黙の型変換 | ❌ | P2。唯一の例外は int→float の拡大のみ |

## 変数

| 機能 | 状態 | 備考 |
|---|---|---|
| 宣言 `:=`(**デフォルト不変**) | ✅ | 2026-07-17決定・実装済み: `x := 10` は不変束縛。
      シャドーイング禁止と合わせ「一つの名前=一つの値」が保証される。生成JSは `const` |
| 型注釈つき宣言 `x: T = v` | ✅ | 2026-07-18実装。推論が効かない/効かせたくないときに型を明示。
      `mut best: string \| none = none`(不在から積み上げ)、`xs: Todo[] = []`(空配列)。
      `mut` と併用可。実験(第3回)で判明した「不在アキュムレータが書けない」問題の解決 |
| 可変宣言 `mut` | ✅ | 2026-07-17決定・実装済み。再代入したい変数だけ `mut count := 0`(生成JSは `let`)。
      レビュー・AIは mut だけ注視すればよい。Rustの `let mut` 相当。多値宣言では全名前に適用 |
| 再代入 `=` / `++` `--` | ✅ | `mut` で宣言した変数のみ(型が合うもののみ)。違反時は
      `'x' is immutable — declare it with 'mut' to allow reassignment` と修正方法つきで報告 |
| 関数引数の再代入 | ❌ | 2026-07-17決定・実装済み。引数は常に不変束縛 |
| 深い不変性(配列・mapの中身まで) | ❓ | v1の `mut` は束縛(再代入)のみ守る=JSのconst相当。
      `nums := [1,2]` への `push(nums, 3)` は通る。readonly型の導入は将来検討 |
| ブランク識別子 `_` | ✅ | 値を意図的に捨てる |
| 未初期化の宣言 | ❌ | `var x int` 的なものは無い。空値の発生源を断つ |
| シャドーイング禁止 | ✅ | 2026-07-17決定・実装済み(テストあり)。`:=` は外側スコープ(関数名含む)に
      同名があるとエラー。更新したいなら `=` を使う。ループ変数の兄弟スコープ再利用には影響なし |
| 定数宣言 const | ❌ | 2026-07-17決定。`:=`/`=`の2動詞で足りており、追加は宣言方法を2つにしP1に反する |

## 関数

| 機能 | 状態 | 備考 |
|---|---|---|
| 関数宣言 `fn` | ✅ | `fn add(a: int, b: int) int { ... }` |
| 多値戻り | ❌ | 2026-07-17廃止・撤去済み。`(int, error)` や `return a, b` は「union型を使え」という
      誘導つきコンパイルエラーになる。複数の値を返したいときは struct で返す |
| 無名関数・クロージャ | ✅ | `double := fn(x: int) int { return x * 2 }` |
| デフォルト引数 | ❌ | 2026-07-17決定。「省略していいか」がAIにとって曖昧さの種になる(P2)。関数シグネチャは常に完全にする |
| オーバーロード | ❌ | P1: 同名関数は1つ(ただしメソッドはstructごとに名前空間が別なので同名可。下記参照) |
| **structメソッド**(Goスタイルのレシーバ構文) | ✅ | 2026-07-18実装。`fn (t: Todo) render() string { ... }`。
      内部的には第1引数にレシーバを取る普通の関数(vtable・継承・インターフェース無し)。
      **名前空間は自由関数と完全分離**——`render(t)`は呼べず`t.render()`のみ(P1: 呼び方は常に一つ)。
      よってstruct種類が増えても同名メソッド(`User.describe()`と`Order.describe()`)が衝突しない。
      レシーバはstruct限定(int等プリミティブは不可)。構造体は参照値のため
      Goの値/ポインタレシーバの区別は不要(既存のstruct参照値セマンティクスをそのまま継承)。
      連鎖呼び出し `todos[0].complete().render()` が左から右へ自然に読める |
| 関数型注釈 `fn(int, string) bool` | ✅ | 2026-07-19実装(批評F-1前半)。**ユーザー定義の高階関数が書ける**ようになった
      (`fn apply(f: fn(int) int, x: int) int` / `fn retryInt(f: fn() int \| error, n: int)`)。
      引数・戻り値・structフィールド・型注釈つき宣言・chan/map/配列要素、型の書ける場所すべてでOK。
      読みは関数宣言と同じ(`fn(int) int \| error` のunionは**戻り値側**に束縛。関数自体を
      unionに入れるなら括弧: `(fn(int) int) \| none`)。パラメータ名は書かない(書いたら誘導エラー)。
      **単相のみ**(ジェネリクス`fn<T>(T) T`は未実装 — F-1後半として別途討議) |

## エラー処理

| 機能 | 状態 | 備考 |
|---|---|---|
| 失敗の表現 `T \| error` | ✅ | 2026-07-17実装。`fn divide(a: int, b: int) int \| error`。
      `is error` で絞り込んで処理。`(T, error)` 多値戻りは撤去済み |
| `?` エラー/none伝播 | ✅ | 2026-07-17実装(`!`として)、**2026-07-19に`?`へ改名**(批評F-4決着:
      後置`!`の学習データはSwift/Kotlin=crash・TS=no-opで負の転移。Rustの`?`は意味が完全一致)。
      `v := f()?` — none/error なら呼び出し元へ即return。旧`!`は誘導つきパースエラー。
      戻り値型に失敗メンバーが無い関数内で使うと修正方法つきコンパイルエラー |
| `? "文脈"` コンテキスト付き伝播 | ✅ | 2026-07-19実装(批評F-2前半)。`toInt(s) ? "line ${i}: bad"` —
      失敗時だけ文脈を評価し、`error("文脈: 元メッセージ")` として伝播(**noneも `error("文脈")` に
      昇格**するので、不在に理由を付けられる)。文脈は文字列リテラル(補間可)のみ。
      戻り値型には `error` が必要 |
| `or` フォールバック | ✅ | 2026-07-17実装、**2026-07-19にGo式の明示性へ変更**(批評F-5決着):
      素の `or 式` は **none専用**(`m[k] or 0` は今まで通り)。errorを含むunionには束縛形が必須 —
      `or e => 式`(失敗値を使う)/ `or _ => 式`(意図的に捨てる。Goの `v, _ :=` と同様、
      握りつぶしが字面に残りgrepで監査できる)。素の`or`でerrorを吸おうとすると
      `'or' would silently discard an error — bind it ...` の誘導エラー。右辺は失敗時のみ評価 |
| `== none` の禁止 | ✅ | narrowing が効く `is none` に一本化(P1)。`==` で書くと誘導エラー |
| `error("msg")` でエラー生成 | ✅ | |
| 例外 try / catch / throw | ❌ | P2: エラーは全てシグネチャに現れる |
| `panic()` / `recover` | ❌ | requirements 5.5。故障時はランタイムが即停止 |
| `must()` | ❌ | panic方針の系 |

## 並行処理

| 機能 | 状態 | 備考 |
|---|---|---|
| `spawn` 式(起動して受取口を返す) | ✅ | 2026-07-18実装(`go` から改名)。`task := spawn f(x)` → `<-task` で結果を受け取る。
      戻り値なしの関数は文として起動のみ。引数は spawn 時点で評価(Goと同じ)。
      **2026-07-18追加(2段スコープ設計)**: spawn は**囲む関数が所有**し、関数を抜けるとき
      (早期returnでも)暗黙に待たれる。**発射しっぱなしのタスク(goroutineリーク)が構文的に不可能** |
| `detach` 式(プログラム所有のバックグラウンドタスク) | ✅ | 2026-07-18実装。構文・受取口は spawn と対称。
      呼び出し元は待たずに戻れて、タスクはプログラム終了までに完了する(JSイベントループが自然に保証)。
      メール送信・ログ等「関数より長生きすべき仕事」の明示的エスケープハッチ。`mut`同様grepで一覧できる。
      常駐ワーカー(チャネル受信ループ)は spawn ではなく detach で起動する |
| チャネル `chan<T>` / `<-` / `ch <- v` | ✅ | E-1討議の結果、channel は言語の顔として維持
      (廃止して構造化並行のみに寄せる MoonBit 案は不採用。タスクの寿命は2段スコープで構造化済み) |
| `wait` ブロック(早めに待つ) | ✅ | 2026-07-18実装。ブロック内で spawn したタスクの完了を待ってから抜ける。
      関数出口の暗黙waitより**早い時点**で待ちたいときに使う。早期 return でも待ち漏れない(try/finally 保証) |
| チャネル容量指定・同期送信 | ✅ | 2026-07-18実装。`chan<T>()`(引数無し)は無制限バッファ(既定・非破壊)、
      `chan<T>(0)`は真の同期(受信者が現れるまで送信ブロック)、`chan<T>(n)`はバッファn(**Go互換の
      本物のブロッキング送信** — panicによる代替案ではなく`await`可能な真のブロックとして実装) |
| チャネルの close / `T \| closed` | ✅ | 2026-07-18実装。`<-ch`は常に`T \| closed`を返す(mapの`V \| none`と同じ理由:
      closeされうることを型で強制)。`close(ch)`で終端を宣言。二重close・close済みへの送信はpanic。
      `closed`は`none`/`error`とは別概念で`?`/`or`の対象には含めない(`is closed`/matchで絞り込む) |
| `select` 式 | ✅ | 2026-07-18実装。`select { v := <-ch1 => ...  v := <-ch2 => ...  _ => ... }`。
      見た目はmatchを踏襲しつつ独立構文(パターンが型ではなく「どの受信操作が先に終わったか」のため)。
      複数同時準備なら擬似ランダムに選択(Goと同じくcase飢餓を防ぐ)。`_`は非ブロッキング用の任意default。
      受信ケースのみ対応(送信ケースは無し。Meshのchannelは既定無制限バッファのため送信不可はほぼ無意味) |
| マルチスレッド並列 | ❌ | JSイベントループ上の並行。並列が要る計算は将来のWASMターゲット検討事項 |

## 制御構文

| 機能 | 状態 | 備考 |
|---|---|---|
| `if` / `else`(条件の括弧なし) | ✅ | 条件は bool 必須 |
| `for` 3形態(C風 / while相当 / 無限) | ✅ | C風forのヘッダで宣言した変数だけは暗黙に可変
      (2026-07-17決定。デフォルト不変の唯一の構造的例外) |
| `break` / `continue` | ✅ | |
| `while` / `do-while` | ❌ | for に統一(P1) |
| `match` 式 | ✅ | 2026-07-17実装(Rust風)。`match r { error => 式  int => 式 }` で値を返す。
      アームは `パターン => 式`(改行区切り、カンマで複数パターン、`_` は最後のみ)。
      **網羅性検査つき**: 足りないメンバーを名指しでエラー。アーム内は対象変数が絞り込まれる。
      パターンは型(none/error/int/...)と文字列リテラル(`"active" => ...`)に対応。
      2026-07-19: 判別可能union用の部分構造パターン `{ kind: "ok" }` にも対応(書いたフィールドが
      一致するメンバーへ絞り込む。フィールド束縛はせず、絞り込み後は普通のフィールドアクセスを使う) |
| `switch` | ❌ | 2026-07-17決定。match式に一本化(分岐は if と match の2つだけ、P1) |
| `for range`(配列・mapの走査) | ✅ | 2026-07-18実装。常に完全形 `for i, v := range arr` / `for k, v := range m`
      (単変数形は無し=Goの「単変数は添字」の罠を排除。不要な方は `_`)。例外は `for i := range 10` のみ。
      ループ変数は反復ごとに新しい不変束縛 |
| セミコロン | ❌ | 行末に自動挿入(書かない) |

## モジュール・相互運用

| 機能 | 状態 | 備考 |
|---|---|---|
| 複数ファイル(import / export) | ✅ | 2026-07-19実装。**パッケージ = ディレクトリ**(ディレクトリ名がそのまま
      パッケージ名。`package`宣言は無い)。パッケージ内の複数`.mesh`ファイルはimport不要の
      フラット名前空間。`export`を付けたトップレベルの fn/struct/type だけが外から見える
      (メソッドは個別exportせず、structのexportに従う)。使う側は `import "mathutil"` の後、
      常に `mathutil.add(1, 2)` / `mathutil.Point{...}` / 型位置 `mathutil.Point` と修飾する
      (P1: 呼び方は一つ)。未exportアクセス・未知パッケージ・import循環・自己importは
      コンパイルエラー。エントリプログラムはエントリファイル1つ(分割はパッケージ単位で行う)。
      **v1の制限**: パスは単一ディレクトリ名のみ(`"a/b"`ネスト不可)、`"mesh/..."`は将来の
      標準ライブラリ用に予約。コンパイラ本体はfs非依存(`compileModules`にソース列を渡す設計)で、
      ファイル読み込みはCLIの仕事(プレイグラウンドは従来通り単一ファイル) |
| 標準ライブラリの層分け(C-6) | 🔜 | モジュールシステム(上)が土台として完成。環境別モジュール
      (`mesh/http`=サーバー、`mesh/dom`=ブラウザ等)の中身とimportからの環境自動推定は、
      Q3(フロントエンドの形)の討議とセットで次段階 |
| npm ライブラリの呼び出し | ❓ | 未決Q2。直接importか、ラッパー経由か |
| JS例外の境界変換 | 🔜 | 境界で catch して `T \| error` に変換 |
| null / `undefined` の正規化 | 🔜 | JS境界でどちらも `none` に変換(Mesh内には汎用空値が無いため) |
