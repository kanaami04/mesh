## 発見した穴の分類

### A. 実装の穴(機能が無い/不足)
- **空の型付き配列**(第1回)→ 解消: `Todo[]{}` / `int[]{...}` / 多次元 `int[][]{}` を実装
- **不在から始まる可変アキュムレータ `mut best := none`**(第3回)→ **解消(2026-07-18)**。
  型注釈つき宣言 `mut best: string | none = none` を実装(対策案a)。あわせて空配列も
  `xs: Todo[] = []` で書けるようになった(`any[]` を型付き配列へ入れられるよう assignable を緩和)。
  つまり第1回の空配列問題と第3回の不在アキュムレータ問題が、型注釈宣言1つで両方解決した。

### B. カードの記述漏れ(実装は正しく動く)→ 追記で解消済み
- 配列/map/struct/chan は参照値・関数に渡すと共有(第2回)
- グローバル変数なし、共有状態は引数で渡す(第2回)
- `:=` はリテラル型を string に widening、`mut s := "a"` は再代入可(第2回)
- 前置 `!`(否定)が存在する。後置 `x!`(伝播)とは別物(第2回)※機能はあったが使われず冗長化していた
- `print` はスペース区切り+末尾改行(第2回)
- error のメッセージは `is error` で絞ってから `${e}` で補間(第1回)
- push は none を返す・値として使えない(第1回)
- `len()` は map のキー数にも使える(第3回)
- map の読み書き更新イディオム `m[k] = (m[k] or 0) + 1`(第3回、+= も comma-ok も無いので明示が要る)
- `if x is error { ... } else { ... }` で else 側も自動的に絞り込まれる実例がカードに無い(第5回)。
  早期return形しか載っておらず、AIは「たぶん対称なはず」と推測で書いて正解した(実装は元々対応済み)。
  **→ 2026-07-19、カードに追記して解消済み**
- `for` 内の `break` が `is` narrowingを続けさせる実例がカードに無い(第6回)。`if op is none { break }`
  の後で `op` を普通の値として使えるかは未確認のまま推測で書いて正解(returnと同じ扱いで実装済み)
- `is` を `||`/`&&` と組み合わせた複合条件でnarrowingが効くかカードに書いていない(第6回)。
  AIは確証が持てず2つのif文に分けて安全側に倒した(実際に効くかは未検証 — 要調査)

### C. 「機能はあるが今回は使わなかった」— 穴ではなく設計判断の記録
- **判別可能union**(第5回): カードの説明は理解された上で、「フラットな数値列に分岐する形が無い
  (数値↔演算子が交互に並ぶだけで、下流で区別すべき別形状が無い)」ため不使用と判断。
  AST的な分岐(単項/二項ノード、将来の括弧対応など)が要るタスクなら使う価値があると明言。
  → 機能の存在と、その機能がタスクに適切かは別問題という健全な判断力の実証
- **判別可能union(再挑戦)**(第6回): AST分岐が要る題材で意図的に使わせたところ、自己参照の壁
  (`type alias cycle`)に当たって断念。カードが提案する回避策(名前付き再帰struct)は試されず、
  木を持たない直接評価へアーキテクチャごと変更する道を選んだ——回避策の実例が薄いカードの記述と、
  「全フィールド必須」の制約下で疑似optionalフィールドのstructを書く面倒さが、回避策を選びにくく
  している可能性がある
