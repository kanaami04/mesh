# Mesh 要件定義書(ドラフト v1)

> 状態: 討議中のドラフト。この文書がプロジェクトの「正」であり、
> 実装(v0含む)はこの文書に従う。矛盾があれば文書が勝ち、実装を直す。

## 1. ビジョン

**AIが書き、人間が読む時代のための言語。**

コードの主な書き手がAIコーディングエージェントになりつつある。しかし既存の言語は
すべて「人間が書くこと」を前提に設計されている。Mesh は前提を逆転させる:

- **書くのは主にAI** — 一発で正しく書けて、間違えてもすぐ自己修正できること
- **読むのは主に人間(チーム)** — 誰が(何が)書いても同じ形になり、レビューしやすいこと
- **フロントエンドもバックエンドも1言語** — AIが型を共有したまま全体を実装できること

## 2. 課題認識(なぜ既存言語ではだめか)

| 言語 | 良さ | AI・チーム開発での問題 |
|---|---|---|
| TypeScript | 型が強い。フロント/バック両方書ける | 同じことを書く方法が多すぎる(型パズル、クラス/関数/OOP/FP混在)。AIの出力がプロジェクトごとにブレる。tsconfigの組み合わせで意味が変わる |
| Go | シンプル。誰が書いても同じ形。エラーが明示的 | ブラウザで動かない(フロント不可)。型表現が弱い場面がある |
| Python/JS | AIの学習データが最多 | 動的型のため、AIの間違い(幻覚)がコンパイル時に捕まらず実行時に漏れる |

**核心の仮説: 言語の設計がAIのコード生成の正答率を左右する。**
選択肢が少なく、間違いが型で即座に捕まり、エラーメッセージが機械可読な言語は、
AIエージェントの「生成→検証→修正」ループを速く・確実にする。

## 3. ターゲットユーザー

1. **AIコーディングエージェント**(主たる書き手)— Claude Code のようなツール
2. **人間の開発チーム**(主たる読み手・レビュアー)— 仕様を決め、AIの出力をレビューする
3. 人間が直接書くことも当然できるが、設計上の最適化対象は 1→2 の順

## 4. 設計原則

実装のあらゆる判断はこの6原則に照らして決める。番号は優先順位ではなく識別子。

### P1. 一つのことを書く方法は一つ(Go由来)
同じ処理は誰が(何が)書いてもほぼ同じコードになる。
AIの出力が安定し、人間のレビューコストが下がる。
→ ループはforのみ / クラス継承なし / フォーマッタに設定なし

### P2. 暗黙より明示(Go由来)
隠れた制御フロー・暗黙の変換を持たない。関数のシグネチャを見れば
「何が起き得るか」が全部わかる。AIは局所情報だけで正しい判断ができる。
→ エラーは戻り値 `(T, error)`(例外なし) / 暗黙の型変換なし / グローバルな魔法なし

### P3. 局所で読める
ファイル(関数)単体で意味が確定する。遠くの設定・デコレータ・継承階層に
意味を左右されない。**AIのコンテキストウィンドウに収まる範囲で正しく理解できる**ことは、
人間にとっての読みやすさと同じ方向を向いている。

### P4. コンパイラはAIの相棒(Mesh独自の重点)
AIエージェントは「コンパイルエラーを読んで直す」ループを高速に回す。
だからエラーメッセージは人間向けの親切さに加えて**機械可読**であること。
→ 位置+原因+修正ヒントを必ず含む / `--json` で構造化出力 / 1回の実行で複数エラーを報告 / コンパイルは高速

### P5. 新規性予算(originality budget)
Meshは独自の書き方を持つ。ただし「新しさ」はコストのかかる予算として扱う。
まったく新しい記号体系や意味論はAIにとって学習データゼロで正答率を下げるが、
既存要素(TS/Go)の再結合や、規則が単純で例外のない小さな独自構文は低コストで済む。

新構文を入れるときの判断基準:
1. **対訳1行テスト**: 「TS/Goでいうと○○」と1行で説明できるか
2. **例外ゼロ**: その規則に例外・特殊ケースがないか(一貫していればAIは外挿できる)
3. **家賃を払うか**: TS/Goのどちらにもない価値(並行処理・エラー処理・型共有まわり)を生むか

また、新規性のコストは診断品質(P4)と言語カード(5.3)で相殺する —
コンパイラが「こう直す」と教えられる構文なら、AIの修正ループ1往復で回収できる。

### P6. フルスタック一体(TS由来)
1つの型定義をフロントエンドとバックエンドで共有できる。
AIがAPI境界をまたいで一貫した実装を一度に生成できる。
→ JavaScriptへコンパイル / 将来: サーバー/クライアント間の型付きAPI呼び出しを言語またはstdlibで支援

## 5. 機能要件

### 5.1 言語コア(v0で実装済み)
- [x] 静的型付け+型推論(`:=`)— P2, P5
- [x] 明示的エラーハンドリング `(T, error)` / `if err != nil` — P2
      ※ union路線決定(2026-07-17)により `T | error` + narrowing へ移行予定。多値戻りは廃止予定
- [x] goroutine/channel風の並行処理(`chan<T>` / `<-`)— P5(Goの既知イディオム)
      ※ v0では `go` キーワードで実装。`spawn`(受取口を返す式)への改名・拡張が決定済み(5.2)
- [x] 小さい文法: for一種類・例外なし・セミコロン自動挿入 — P1
- [x] JavaScriptへのコンパイル — P6

### 5.2 言語コア(未実装・要件として確定)
- [ ] `!` エラー伝播・`or` フォールバック(2026-07-17採用)— P2, P5
- [ ] `spawn` 式: 並行起動して受取口を返す(2026-07-17採用、旧 `go` を改名・拡張)— P5, P6
- [ ] `wait` ブロック: 構造化並行、中で起動したタスクの完了を待つ(2026-07-17採用)— P1, P2
- [ ] `select` 文: 複数チャネルの待ち合わせ(2026-07-17採用、実装順は後段)— P5
- [ ] struct による型定義(フロント/バック共有の要)— P6
      構文(2026-07-17改訂): `struct User { name: string }`。union・別名は `type X = ...` で分離。
      型付けは**構造的**(形が同じなら同じ型。TS方式。JSON/API境界と透過的)
- [ ] union 型: `type Status = "active" | "banned"`(2026-07-17採用、
      まず文字列リテラル union から段階的に)— P2, P6
- [ ] 文字列補間: 通常の `"` 内で `${式}`(2026-07-17採用)— P5
- [ ] デフォルト不変 + `mut`(2026-07-17決定)— P2, P3
      `:=` は不変束縛。再代入できるのは `mut x := 0` と宣言した変数のみ。引数は常に不変。
      C風forヘッダ変数だけ暗黙可変。シャドーイング禁止と合わせ「一つの名前=一つの値」を保証。
      v1は束縛レベルの不変性(深い不変性は将来検討)
- [ ] 【背骨】union路線(2026-07-17決定。同日の「空値の一本化(nil→null改名)」を発展的に置換)— P1, P2, P4
      **実装済み(2026-07-17)**: union型・`T | none`・`T | error`・`is` narrowing・`!`・`or`・
      match式(網羅性検査つき)・文字列リテラル型・`type` 宣言・nil/多値戻りの撤去
      不在は `T | none`、失敗は `T | error` の union で表現し、narrowing / match で処理する。
      **絞り込み前の使用・処理漏れはコンパイルエラー**(核心仮説「AIのミスを機械的に弾く」の本体)。
      汎用の null 値は存在しない / `undefined` も存在しない / 未初期化宣言は不可 /
      `(T, error)` 多値戻りは廃止 / `!`・`or` は union 上で再定義 / map 読みは `V | none` /
      JS境界では null・undefined を `none` に、例外を `T | error` に正規化
- [ ] map型 `map<K, V>`: 読み取りは comma-ok 形式のみ(2026-07-17採用)— P2, P4
- [ ] for range: 2変数の完全形のみ+`range n`(2026-07-17採用、Goの単変数の罠を排除)— P1, P4
- [ ] match式: Rust風 `パターン => 式`、union型の網羅性検査つき(2026-07-17採用。
      switchは入れず、分岐は if と match に一本化)— P1, P2, P5
- [ ] モジュールシステム(import/export)— 複数ファイル構成に必須
- [ ] JS/npm との相互運用(既存エコシステムを捨てない)— P6

### 5.3 ツールチェーン(AI-first の本丸)
- [ ] エラーメッセージ規約: 位置+原因+「どう直すか」ヒントを必ず含む — P4
- [x] `mesh check --json`: 診断の構造化出力(エージェントが直接パースする)— P4(2026-07-18実装)
- [ ] エラー復帰: 1回の実行で複数のエラーをまとめて報告 — P4
- [ ] `mesh fmt`: 設定オプションのないフォーマッタ — P1
- [x] `mesh test`: テストランナー内蔵(AIの検証ループを1コマンドに)— P4(2026-07-20実装。F-15)
- [x] **言語カード**(2026-07-18実装): AIのコンテキストに入れる前提で設計した圧縮仕様書
      (全構文+TS/Go対訳例+頻出エラーの直し方)。`mesh card` で出力でき、
      CLAUDE.md 等にそのまま貼れる形式にする — P5

### 5.4 やらないこと(スコープ外を明文化する)
- 例外・try/catch(P2に反する)
- ユーザーが呼べる `panic()` / `recover`(5.5 のパニック方針を参照)
- 汎用の空値(null / nil / undefined)— 不在は `T | none` の union で型に現す(2026-07-17 union路線)
- `(T, error)` 多値戻り — `T | error` に置換(2026-07-17 union路線)
- 未初期化の変数宣言(宣言は必ず値と同時)
- クラス継承・メソッドオーバーロード(P1に反する)
- マクロ・演算子オーバーロード・デコレータ(P3に反する)
- 設定可能なコンパイラオプションによる言語仕様の変化(tsconfig問題の再発防止、P1)
- 独自VM・ネイティブコンパイル(当面。JSエコシステムに乗る方が P6 に効く)

### 5.5 パニック方針(故障時の挙動)— 2026-07-17 決定

「予期される失敗」はすべて `(T, error)` 戻り値で扱う(P2)。それとは別の
「バグによる故障」については以下の通り:

| 項目 | 方針 |
|---|---|
| ランタイム故障(配列範囲外・ゼロ除算・nil参照など) | 即時停止。診断はP4品質(位置+原因+ヒント)。JSの意味論(`undefined` / `Infinity`)をMeshの世界に漏らさない |
| ユーザー向け `panic()` 関数 | 提供しない。「起こり得ないはず」はエラー戻りで表明する。機能が存在しなければAIも「とりあえずpanic」を書けない(P1) |
| `recover` | 提供しない。入れた瞬間にエラーが2系統(戻り値+飛んでくる例外)になり例外システムが誕生してしまう(P2) |
| 障害分離(サーバーが1リクエストのバグで全体を落とさない等) | stdlib/ランタイムのJS実装内部の仕事とし、言語からは見せない(Go の net/http が内部で recover するのと同じ構図) |
| JS相互運用で飛んでくる例外 | 境界で catch して `(T, error)` に変換する |
| ランタイム内部の `__panic` | 残す。上記「即時停止」を実装する内部部品(防波堤)であり、Meshコードからは呼べない |

背景: TS/JSはバグと予期される失敗を両方 throw で扱い、シグネチャから失敗が
見えない・catchがバグを飲み込むという問題を抱えた。Go/Rustは分離を進めた。
Meshは分離をさらに進め、復帰機構ゼロの設計を取る。

## 6. 非機能要件

| 項目 | 要件 |
|---|---|
| コンパイル速度 | 1万行を1秒以内(目安)。AIの修正ループを待たせない |
| 診断品質 | すべてのエラーに位置・原因・修正ヒント。将来 `--json` |
| 決定性 | 同じ入力→常に同じ出力。フォーマットも一意 |
| 出力JSの可読性 | 生成コードは人間がデバッグできる素直なJS |
| 依存 | コンパイラ本体はランタイム依存ゼロを維持 |

## 7. 成功基準(どうなったら「AIに実装しやすい」と言えるか)

測定可能な形で定義する:

1. **一発成功率**: 同じ仕様(例: TODOアプリのAPI+画面)をAIに Mesh / TypeScript / Go で
   書かせ、初回コンパイル成功率・初回テスト通過率を比較して Mesh が上回る
2. **修正ループ数**: コンパイルエラーからの自己修正が平均何往復で収束するか
3. **レビュー容易性**: 同じ仕様を複数回生成したときの出力の一貫性(diffの小ささ)

v0(2026-07-17実装)はこの要件定義より先に作られたが、方向性の矛盾はない。
5.1 はすべて充足。主なギャップは:

- ~~ランタイム故障がJSの意味論のまま素通りしている~~ → **解消(2026-07-17実装)**。

- フォーマッタなし → 5.3
