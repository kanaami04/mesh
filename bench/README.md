# 対TS/Go比較ベンチマーク(F-12)

> 核心仮説「**言語設計がAIの正答率を左右する**」の効果測定。
> カード実験(docs/card-experiments.md)の進化形: 同一タスクを Mesh / TypeScript / Go で
> 白紙のAIエージェントに書かせ、成功率と修正往復数を機械的に比較する。
> 結果の記録は docs/benchmark-2026-07.md。

## 計測プロトコル(1セル = 1タスク × 1言語 × 1試行)

1. **白紙のサブエージェント**(被験体: Sonnet)に「タスク仕様 + 言語指定」を渡す
   - Mesh: 言語カード(`mesh card` の出力)を同梱する。カードはMeshの資産の一部
     (「カード=検証済み圧縮仕様」が言語の提供物)なので、同梱した状態が公平な比較
   - TS / Go: カード無し。学習データが彼らの資産
   - 被験体はコンパイラ・処理系を**実行できない**(書くだけ)。往復数を正確に数えるため
   - 被験体はリポジトリ内のファイルを読めない(Meshの場合、カードファイル1つだけ例外)
2. オーケストレータ(親エージェント)がコンパイル検査を実行:
   - Mesh: `bun src/cli.ts check <file>`
   - TS: `bunx tsc --noEmit --strict --ignoreConfig <file>`(`--ignoreConfig`はパイロットで
     判明したハーネス側の必須フラグ — リポジトリの`tsconfig.json`と競合するため)
   - Go: `go build -o /dev/null <file>`
3. エラーがあれば**生のコンパイラ出力をそのまま**被験体に返して修正させる(1往復と数える)。
   整形・ヒント追加はしない — 「エラーメッセージが修正方法を含む」というMeshの主張自体が
   計測対象のため
4. コンパイルが通ったら実行し、期待stdout(バイト一致)と照合:
   - Mesh: `bun src/cli.ts run <file>` / TS: `bun <file>` / Go: `go run <file>`
5. 出力が違う場合は expected / got の差分を返して修正させる(これも1往復)
6. **上限6往復**で打ち切り(カード実験第1回の実測最大値に合わせる)

## 記録する指標(セルごと)

| 指標 | 定義 |
|---|---|
| 一発コンパイル | 初回提出がコンパイル検査を通ったか |
| 一発成功 | 初回提出がコンパイル+期待出力まで一致したか |
| 往復数 | 全緑になるまでの修正回数(0 = 一発成功) |
| 最終成功 | 6往復以内に全緑になったか |
| 行数(参考) | 最終コードの行数(空行・コメント除く) |

## 公平性の担保

- 同一モデル・同一プロンプトテンプレート・同一上限往復数
- タスク仕様は言語中立の英語(3言語とも学習データは英語優位)で、期待stdoutを明記
- 採点は期待stdoutとのバイト一致のみ(人間の裁量を挟まない)
- フィードバックは生のコンパイラ/差分出力のみ(追加のヒントを与えない)
